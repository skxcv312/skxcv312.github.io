프로젝트 개요

주요 기능 및 내 역할 (My Role)

  • Spring Boot 및 FastAPI 백엔드 개발
  • Chrome Extension 기반 검색어, 상품 조회, 카테고리 탐색, 장바구니, 장소 검색, 클릭, 체류 시간, 스크롤 데이터 수집
  • 검색, 상품 조회, 장바구니, 장소 클릭 등 행동 로그를 유형별로 분류하고, 행동 중요도와 체류 시간 기준을 가중치로 반영해 사용자 관심 키워드 점수를 산출
  • 관심 키워드 그룹화와 네이버 쇼핑·지역 검색 API를 활용해 맞춤형 상품과 장소 추천
  • 관심 점수를 기반으로 영상 검색어를 생성하고, YouTube 영상 정보와 자막/설명 요약 제공

트러블슈팅 로깅

웹 행동 로그 수집 파이프라인 구축

Root Cause:

  • 사용자의 검색, 상품 조회, 장바구니, 장소 탐색 기록이 무신사, 쿠팡, 네이버 등 여러 웹사이트에 흩어져 있어 관심사 분석에 바로 활용하기 어려웠음
  • 사이트마다 DOM 구조와 URL 패턴이 달라 동일한 방식으로 상품명, 가격, 카테고리, 검색어, 장소 정보를 추출할 수 없었음
  • SPA 방식 페이지에서는 URL이 변경되어도 새로고침이 발생하지 않아 로그 누락 가능성이 있었음
  • 단순 페이지 방문 기록만으로는 사용자의 실제 관심도를 판단하기 어려워 체류 시간, 스크롤, 클릭 행동까지 함께 수집할 필요가 있었음

로그 수집

FLOWChrome Extension 로그 수집
CLASSIFY로그 타입 분류
OBJECT행동 로그 생성
APIFastAPI 로그 전송
STOREMySQL 행동 로그 적재
UPDATEevent 갱신

로그 검증

LOG상세 로그 저장
BACKUPlocal storage 저장
ACTION테스트 데이터 검증
STOREJSON 다운로드

Solution Steps:

  1. Chrome Extension Manifest V3 기반으로 content script와 background service worker를 구성
  2. 무신사, 쿠팡, 네이버 사이트별 핸들러를 분리해 상품/검색/카테고리/장바구니/장소 페이지 데이터를 각각 추출
  3. 사이트별로 다른 데이터를 url, timestamp, view, clickTracking, details를 포함한 공통 로그 포맷으로 표준화
  4. viewTracker를 구현해 체류 시간, 총 스크롤 거리, 탭 비활성 시간, 60초 이상 무활동 시간을 반영
  5. MutationObserver와 history.pushState/replaceState 감지를 통해 SPA URL 변경 시에도 로그를 전송하도록 구성
  6. 서버 전송과 동시에 Chrome local storage에 로그를 저장하고 JSON 파일로 다운로드할 수 있게 하여 테스트 및 데이터 검증 편의성 확보

Result:

  • 여러 웹사이트에서 발생하는 사용자 행동을 하나의 공통 로그 구조로 수집할 수 있는 데이터 수집 파이프라인 구축
  • 단순 방문 기록이 아닌 체류 시간, 스크롤, 클릭, 좋아요, 장바구니 등 실제 관심 행동을 분석 가능한 데이터로 전환
  • SPA 페이지 이동 및 탭 종료 상황에서도 로그가 누락되지 않도록 처리하여 수집 안정성 개선
  • 관심 키워드 점수 계산과 개인화 추천 기능의 기반 데이터로 활용 가능하도록 설계

사용자 행동 기반 관심 키워드 점수화

Root Cause:

  • 수집된 로그를 단순 저장하는 것만으로는 사용자가 어떤 키워드에 더 높은 관심을 보였는지 판단하기 어려웠음
  • 검색, 상품 조회, 카테고리 탐색, 장바구니, 장소 검색은 관심 강도가 서로 다르기 때문에 동일한 점수로 처리하면 추천 품질이 떨어질 수 있었음
  • 사용자가 오래 머물거나 많이 스크롤한 페이지와 잠깐 방문한 페이지를 동일하게 평가하면 관심도 계산이 왜곡될 수 있었음
  • 하나의 사용자가 여러 이메일 연동 계정에서 로그를 생성할 수 있어 사용자 단위로 관심 키워드를 통합할 필요가 있었음

키워드 분석

TRIGGER추천 요청 발생
QUERYMySQL 로그 조회
WEIGHT로그 타입별 가중치 적용
CALC체류·스크롤·클릭 점수 반영

통합 분석

COLLECT사용자 이메일별 점수 수집
MERGE사용자 단위 통합
SUM중복 키워드 합산
SORT점수 기준 정렬
EXTRACT상위 키워드 추출
REDISRedis 관심 점수 저장

Solution Steps:

  1. 검색, 상품, 카테고리, 장바구니, 장소 로그에 서로 다른 기본 가중치를 부여하는 LogActionWeight 설계
  2. 체류 시간과 총 스크롤 값을 추가 가중치로 반영해 적극적으로 탐색한 페이지의 키워드 점수가 더 높아지도록 구성
  3. WeightingInterest 추상 클래스로 전략 패턴을 만들고, 상품/검색/카테고리/장바구니별 관심도 계산 로직을 분리
  4. 상품 로그에서는 카테고리, 성별/시즌 정보, 좋아요 여부, 장바구니 클릭, 상품정보 더보기 클릭 등을 추가 점수로 반영
  5. 장바구니 로그에서는 상품 수량과 최근 장바구니 이벤트를 반영해 구매 의도에 가까운 행동의 점수를 높게 계산
  6. 장소 로그에서는 검색어, 주소 지역, 장소 카테고리를 분리해 장소 추천에 활용 가능한 키워드로 변환
  7. 여러 이메일 연동 계정에서 수집된 로그를 사용자 기준으로 통합하고, 중복 키워드는 점수를 합산해 쇼핑/장소 타입별 관심 키워드로 정렬

Result:

  • 원시 행동 로그를 키워드 + 타입 + 점수 형태의 정량 데이터로 변환하는 관심도 계산 로직 구현
  • 단순 클릭 수가 아니라 행동 유형, 체류 시간, 스크롤, 장바구니, 좋아요 등 실제 관심 강도를 반영하도록 개선
  • 쇼핑 관심 키워드와 장소 관심 키워드를 분리해 상품 추천, 장소 추천, 마이페이지 차트에 재사용 가능한 구조 마련
  • 여러 연결 계정의 로그를 사용자 기준으로 통합해 개인화 추천의 정확도를 높일 수 있는 기반 구축

추천 결과 캐싱 및 재사용

Root Cause:

  • 추천 요청이 들어올 때마다 전체 행동 로그를 다시 분석하면 응답 시간이 길어지고 서버 부하가 증가함
  • 추천 결과 생성 과정에서 OpenAI, Naver API 등 외부 API 호출이 포함되어 있어 반복 요청 시 비용과 지연 시간이 커질 수 있음
  • 사용자의 로그가 거의 변하지 않았는데도 매번 관심 키워드와 추천 결과를 재계산하는 것은 비효율적이었음
  • 반대로 새로운 로그가 일정량 이상 쌓였을 때는 기존 추천 결과를 계속 재사용하면 최신 관심사가 반영되지 않는 문제가 있었음

재계산/갱신

EVENT로그 업데이트
TRIGGER기준값 초과

APIFastAPI 추천 요청
FILTER중복 추천 제외
CACHERedis 결과 저장

Solution Steps:

  1. 관심 키워드 점수 결과를 Redis에 저장하는 KeywordScoreRedisEntity 구조 설계
  2. 추천 응답 결과를 JSON 형태로 Redis에 저장해 동일 사용자의 반복 요청 시 기존 결과를 즉시 반환하도록 구성
  3. 로그가 저장될 때마다 eventUpdateNum을 증가시켜 사용자의 로그 변경 횟수를 추적
  4. 추천 요청 시 Redis에 저장된 관심 키워드가 있고 변경 횟수가 기준값 미만이면 기존 점수와 추천 결과를 재사용
  5. 로그 변경 횟수가 100회 이상이면 기존 추천 응답을 삭제하고 관심 키워드 점수와 추천 결과를 다시 생성
  6. 추천 결과 생성 후 중복 상품/장소 제목을 제거해 사용자에게 반복된 추천이 노출되지 않도록 후처리

Result:

  • 반복 추천 요청에서 불필요한 로그 재분석과 외부 API 호출을 줄여 응답 효율 개선
  • Redis 캐싱을 통해 추천 결과를 빠르게 재사용하면서도, 로그가 일정량 이상 변경되면 최신 관심사를 반영하도록 균형 있게 설계
  • OpenAI/Naver API 호출 비용과 서버 처리 부담을 줄일 수 있는 구조로 개선
  • 관심 키워드 점수와 추천 응답을 분리 저장해 차트, 추천 목록 등 여러 기능에서 재사용 가능한 캐시 구조 확보

YouTube 영상 요약 비용 최적화

Root Cause:

  • YouTube 영상 전체 자막을 AI 요약 모델에 그대로 전달하면 영상 길이에 비례해 입력 토큰 수가 과도하게 증가함
  • 긴 영상일수록 OpenAI API 비용과 응답 시간이 증가하고, 요약 요청 실패 가능성도 높아질 수 있었음
  • 모든 영상에 자막이 제공되는 것이 아니기 때문에 자막이 없는 경우 요약 기능이 중단될 수 있었음
  • 짧은 영상은 정보량이 적어 추천 콘텐츠로서 가치가 낮고, 요약 비용 대비 효율이 떨어질 수 있었음

요약 최적화

TRANSCRIPT자막 데이터 조회
CUT10~600초 필터링

RESPONSE4문장 한국어 반환

Solution Steps:

  1. YouTube Data API를 통해 영상 제목, 채널, 썸네일, 업로드일, 재생 시간 등 기본 메타데이터를 먼저 조회
  2. 영상 길이를 초 단위로 변환한 뒤 180초 미만의 짧은 영상은 추천 대상에서 제외
  3. YouTube Transcript API로 자막을 조회하되, 전체 자막을 사용하지 않고 10초부터 600초까지의 구간만 필터링
  4. 필터링된 자막 텍스트만 OpenAI 요약 모델에 전달해 불필요한 입력 토큰 사용을 줄임
  5. 자막이 없거나 내용이 부족한 경우에는 영상 설명란을 대체 입력으로 사용해 요약 기능이 중단되지 않도록 처리
  6. 요약 모델의 출력 길이를 제한하고, 4문장 이내의 한국어 요약으로 응답하도록 프롬프트를 구성

Result:

  • 전체 자막을 그대로 전달하던 방식에서 필요한 구간만 선별해 전달하는 방식으로 변경하여 AI 입력 토큰 사용량을 절감
  • 긴 영상에서도 요약 요청의 비용과 응답 시간을 줄일 수 있는 구조로 개선
  • 자막이 없는 영상도 설명란을 활용해 요약 실패 가능성을 낮춤
  • 짧은 영상 제외, 자막 구간 제한, 출력 길이 제한을 통해 추천 영상 요약의 품질과 API 사용 효율을 함께 개선
