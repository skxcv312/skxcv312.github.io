MODULE: 03. Key Features
03

핵심 기능 구현

1. 산책 활동 기록

Problem/Role: 산책 종료 시 발생하는 시간, 거리, 휴식, 경로, 참여 반려견 데이터를 이후 통계와 일기에서 재사용 가능한 활동 데이터로 저장

Performance:

  • 산책 시작/종료 시간, 총 거리, 평균 페이스, 휴식 시간을 함께 기록
  • 경로 조회에는 원본 좌표 리스트를 JSON 형태로 보존해 사용하고, 공간 분석과 확장 가능성을 위해 시작 지점, 종료 지점, 전체 경로는 Geometry 기반 데이터로 함께 저장
  • 한 번의 산책에 여러 반려견을 연결할 수 있도록 산책-반려견 매핑 구조를 두고, 요청한 반려견이 실제 로그인 사용자의 반려견인지 검증

2. 반려견별 활동 통계

Problem/Role: 저장된 산책 데이터를 일간, 주간, 월간 통계로 제공해 보호자가 반려견의 활동량을 확인할 수 있도록 구현

Performance:

  • QueryDSL로 날짜별 산책 횟수, 거리, 시간, 휴식 시간을 집계
  • 반려견의 견종 정보는 Enum으로 표준화해 관리하고, 반려견별 기본 정보와 산책 기록을 연결해 이후 활동 통계에서 반려견 단위로 산책량을 해석할 수 있도록 구성
  • 주간/월간 통계는 여러 조회를 병렬로 처리해 응답 시간을 줄이도록 구성

3. 산책 동기부여 알림 기능

Problem/Role: 보호자가 산책을 단순 메모가 아니라 시간, 거리, 경로, 휴식, 함께한 반려견까지 포함한 하나의 활동 데이터로 남길 수 있도록 구성

Performance:

  • @Scheduled 기반 산책 리마인드 자동 발송 흐름을 구성하여, 사용자가 앱에 접속하지 않아도 산책 행동을 유도하는 알림 기능 구현
  • 알림 발송 흐름과 메시지 생성 로직을 분리하여, 알림 종류 추가 시 전체 구조 수정 없이 메시지만 확장할 수 있도록 전략 패턴 설계
  • 알림 수신 동의, 활성 FCM 토큰, 알림 타입 설정을 기준으로 발송 대상을 필터링하고, 대상 조회와 FCM 전송을 배치 단위로 분할 처리하여 대규모 발송 체계 구성
  • 실패 토큰을 삭제하지 않고 비활성 상태로 관리하여, 재등록 시 사용자 기존 알림 설정을 유지한 채 복구할 수 있도록 설계

1000명 단위 배치 조회		 알림 전체 데이터를 정렬 순서대로 순회하는 작업이므로 페이지 번호보다 마지막 조회 기준값을 사용하는 Cursor 방식이 적합 전체 대상자를 한 번에 메모리에 올리지 않고 순차 처리 가능
사용자 수가 증가해도 안정적으로 알림 대상자 조회 가능
산책 경로 데이터가 상세 조회와 위치 기반 분석이라는 서로 다른 목적을 가짐 원본 좌표는 JSON, 공간 데이터는 Point,
LineString으로 분리		 경로 재현에는 원본 좌표가 필요하고, 위치 조회에는 Spatial 타입과 인덱스가 적합 상세 조회와 위치 기반 추천으로 확장 가능한 경로 데이터 구조 확보
사용자별 산책 상태에 따라 매일 개인화 리마인드 알림 발송 필요 Spring Scheduled 선택 DB 조회, 알림 설정, FCM 발송이 기존 도메인 로직과 밀접해 내부 스케줄러가 적합 별도 인프라 없이 배치 조회, 메시지 생성, 실패 토큰 처리를 통합 관리
지역별 날씨 데이터의 외부 API 반복 호출을 줄일 캐시 전략 필요 Caffeine Cache 선택 데이터 규모가 작고 TTL이 짧아 별도 Redis보다 로컬 캐시가 단순하고 효율적 외부 API 호출 감소 및 별도 인프라 없는 경량 캐시 구조 구현
알림 유형마다 메시지 생성 기준은 다르지만, 대상자 조회와 FCM 발송 흐름은 반복됨 대상자 조회/FCM 전송은 공통화하고,
알림별 메시지 생성 규칙만 전략 패턴으로 분리		 대량 조회, 커서 페이징, FCM 배치 발송은 공통 로직으로 유지하고, 알림별 메시지 생성 패턴만 독립적으로 교체 가능 새 알림 유형 추가 시 공통 발송 흐름은 유지하고, 메시지 생성 전략 패턴만 추가해 확장 가능
MODULE: 05. Troubleshooting
05

트러블슈팅 로깅

Problem: 통계 조회 시 월별/주별 데이터를 반복 조회하고, 연관 데이터 조합 과정에서 추가 쿼리가 발생해 응답 지연이 발생

Solution Steps:

  1. 통계 조회 성능 개선은 단순히 QueryDSL을 사용한 것이 아니라, 서비스 레이어에서 조회 횟수와 데이터 조합 방식을 함께 개선
  2. 월간 통계는 월별 반복 조회 대신 기간 전체를 한 번에 조회해 월 단위로 그룹핑
  3. 주간 통계처럼 서로 의존하지 않는 조회는 CompletableFuture로 병렬 실행해, 여러 통계 데이터를 조합하는 API의 응답 지연을 줄일 수 있는 구조로 개선

Final Result/Prevention: 이를 해결하기 위해 기간 전체 데이터를 QueryDSL로 한 번에 조회한 뒤 애플리케이션 레벨에서 월 단위로 그룹핑하도록 변경
또한 서로 의존하지 않는 주간 통계 조회는 CompletableFuture로 병렬 처리해 통계 API 응답 시간을 단축할 수 있는 구조로 개선

Problem: S3는 DB 트랜잭션에 포함되지 않기 때문에, 업로드 성공 후 DB 저장이 실패하면 파일이 S3에 남는 문제 발생

Solution Steps:

  1. S3는 DB 트랜잭션에 포함되지 않는 외부 저장소이므로, DB 저장이 실패해도 이미 업로드된 S3 파일은 자동으로 롤백되지 않는 문제를 원인으로 파악
  2. S3 업로드 성공 직후 업로드된 file key를 S3RollbackEvent에 담아 ApplicationEvent로 등록
  3. DB 트랜잭션이 실패한 경우에만 @TransactionalEventListener의 AFTER_ROLLBACK 단계에서 이벤트 처리

Final Result/Prevention: DB 롤백만으로는 S3 파일이 되돌아가지 않기 때문에, 업로드 성공 시 file key를 S3RollbackEvent로 등록하고, 트랜잭션이 실패한 경우에만 AFTER_ROLLBACK 단계에서 S3 파일을 삭제하도록 보상 트랜잭션을 적용

MODULE: 06. Workflow Automation
06

에이전트 기반 개발 워크플로 개선

△ Existing Issues

  • PR 작성, 커밋 메시지 작성, 코드 리뷰 준비처럼 반복되는 개발 운영 업무가 많음
  • 작업자마다 작성 기준이 달라질 경우 협업 품질이 흔들릴 수 있다고 판단

⚑ Approach

  • Gemini CLI가 프로젝트 컨텍스트를 일관되게 이해할 수 있도록 GEMINI.md를 작성
  • 반복 업무를 하나의 범용 AI에게 맡기기보다 목적에 따라 역할별 에이전트를 분리

Implementation:

code-reviewer 에이전트:

코드 변경 사항을 검토하는 역할로 분리

보안 취약점, 성능 문제, N+1 쿼리 가능성, 예외 처리 방식, 프로젝트 컨벤션 준수 여부를 중점적으로 점검하도록 정의

project-manager 에이전트:

구현 세부사항보다 아키텍처 방향, 기능 우선순위, 기술 선택의 타당성을 검토하는 역할로 분리

복잡한 기능을 구현하기 전 데이터 흐름, 도메인 책임, 확장 가능성을 먼저 점검하는 용도로 활용

PR 작성 컨벤션:

git diff, 커밋 히스토리, 변경된 DTO/Service/Exception 구조를 분석해 PR 초안을 만들 수 있도록 규칙을 정의

Summary, Changes, Test Scenarios 형식을 고정해 PR 설명 품질을 일정하게 유지

커밋 메시지 컨벤션:

변경 파일과 작업 의도를 기준으로 커밋 메시지 초안을 작성할 수 있도록 형식을 정의

type, subject, issue number, 변경 클래스별 상세 내용을 포함하도록 구성

Result

  • 반복적인 문서 작성 시간을 줄이고, PR과 커밋 메시지의 형식을 일관되게 유지
  • 코드 리뷰 시 보안, 성능, 예외 처리, 컨벤션 준수 여부를 빠르게 점검할 수 있는 개발 흐름 구성
