MODULE: 02. Troubleshooting

트러블슈팅 로깅

통계 조회 지연 개선

Root Cause:

  • 통계 API 요청 시마다 일별/주별 산책 데이터를 원본 테이블에서 반복 조회
  • 조회된 데이터를 응답 시점에 다시 날짜별, 주별, 월별로 조합하면서 계산 비용이 커짐
  • 연관 데이터까지 함께 조합하는 과정에서 N+1 쿼리와 응답 지연이 발생

BATCH FLOW

EVENT산책 기록 저장/변경
BATCH통계 배치 실행
FETCH원본 기록 조회
AGG주/월별 집계
STORE집계 결과 저장

API FLOW

REQUEST통계 API 요청
QUERY사전 집계 데이터 조회
MAP응답 조합
END응답 반환

Solution Steps:

  1. 월간 통계 조회 시 월별 데이터를 각각 조회하는 방식이 아닌, 기간 전체 기록 데이터를 한 번에 조회하는 구조로 변경
  2. QueryDSL을 활용해 필요한 산책 기록 데이터를 조건에 맞게 조회하고, 애플리케이션 레벨에서 월 단위로 그룹핑
  3. 배치에서는 @Scheduler를 활용해 필요한 기간의 원본 산책 기록을 한 번에 조회하고, 날짜/주/월 단위로 집계
  4. 주간 통계처럼 서로 의존하지 않는 조회는 CompletableFuture를 활용해 병렬 실행
  5. 통계 계산 로직을 서비스 계층에서 분리해 조회, 집계, 응답 반환 책임을 구분

Result:

  • 기간 전체 데이터를 QueryDSL로 한 번에 조회한 뒤 애플리케이션 레벨에서 월 단위로 그룹핑하도록 변경
  • 통계 계산을 API 요청 시점에서 배치 작업으로 분리하여, API 요청 중 수행되던 원본 데이터 집계 횟수를 감소
  • 서로 의존하지 않는 주간 통계 조회는 CompletableFuture로 병렬 처리해 통계 API 응답 시간을 단축할 수 있는 구조로 개선

서버 오류 공유 개선

Root Cause:

  • 서버 오류가 발생해도 로그를 직접 확인하기 전까지는 팀원이 문제를 즉시 인지하기 어려웠음
  • 오류 상황을 발견한 사람이 별도로 팀원에게 공유해야 해서 대응이 늦어지는 상황 발생
  • Grafana 같은 모니터링 도구를 도입할 수도 있었지만, 개발 당시 팀원들이 해당 도구에 익숙하지 않아 빠르게 적용하기 힘듦
  • 프로젝트 규모상 별도 모니터링 시스템을 구축하기보다, 팀이 이미 사용 중인 Discord를 활용하는 편이 더 빠르고 현실적 판단

오류 전파 Flow

PROBLEM서버 예외 발생
FLOW전역 예외 처리
FLOWERROR 로그 기록
FILTERERROR 레벨 필터
HANDLER비동기 어펜더
ACTIONDiscord 즉시 알림

Solution Steps:

  1. 전역 예외 처리기에서 공통 에러 응답을 생성하고, 예외 메시지와 Stack Trace를 ERROR 로그로 기록
  2. 모든 로그를 전송하면 노이즈가 커질 수 있으므로, ERROR 레벨 이상의 로그만 필터링
  3. 요청 URL, 예외 메시지, 발생 시각 등 원인 파악에 필요한 정보를 Discord Webhook 메시지로 변환
  4. Discord 전송이 API 응답 흐름에 영향을 주지 않도록 비동기 Appender로 구현
  5. 테스트 환경에서는 실제 Discord 전송 대신 로그 이벤트가 ERROR 레벨로 기록되는지 검증
  6. 별도 모니터링 도구 도입 전 단계에서 팀 프로젝트 환경에 맞는 간단한 오류 대응 구조 적용

Result:

  • 별도 모니터링 도구 도입 없이도 팀원이 오류 발생 여부를 빠르게 확인할 수 있는 구조로 개선
  • 서버 로그를 직접 확인해야 하는 과정을 줄이고, 테스트 중 발생한 오류 상황을 팀 채널에서 바로 공유할 수 있도록 구성
  • 프로젝트 환경에 맞는 간단한 오류 대응 구조 구현

FCM 토큰 유효성 개선

Root Cause:

  • FCM 토큰은 앱 재설치, 로그아웃, 기기 변경 등으로 변경되거나 만료됨
  • 서버 DB에는 기존 토큰이 활성 상태로 남아 있어 유효하지 않은 토큰으로 알림을 반복 발송
  • 실패한 토큰을 즉시 제외하지 않으면 이후 알림 발송에서도 같은 실패가 반복되는 문제

복구 흐름

FLOW로그인
FLOW새 FCM 토큰 발급
HANDLER유효성 검증
FLOWDB 재등록
ACTION토픽 재구독

발송 흐름

FLOW활성 토큰 조회
FLOWFCM 일괄 발송
ACTION실패 토큰 비활성화
HANDLER다음 발송 제외

Solution Steps:

  1. 토큰 등록 및 갱신 시 FCM 토큰 유효성 검증 로직 추가
  2. 유효한 토큰만 활성 상태로 저장하고, 검증 실패 토큰은 저장 대상에서 제외
  3. 알림 발송 실패 시 실패 토큰을 수집해 DB에서 isActive=false 상태로 변경
  4. 알림 발송 대상 조회 시 활성 토큰, 알림 수신 동의 여부, 알림 타입 설정을 함께 확인
  5. 토큰을 즉시 삭제하지 않고 비활성화 상태로 관리해 재등록 가능성을 고려
  6. 재등록 시 알림 수신 동의 상태와 알림 타입 설정을 확인한 뒤 필요한 토픽을 다시 구독하도록 처리

Result:

  • FCM 토큰 등록/갱신 시 Firebase dry-run 검증을 적용해 유효성 검증을 거친 토큰만 DB에 저장하도록 개선
  • 발송 실패 토큰은 즉시 비활성화하여 다음 발송 대상에서 제외함으로써 동일한 만료 토큰에 대한 반복 발송을 1회 실패 이후 차단
  • 토큰 상태와 사용자 알림 설정을 기준으로 발송 대상을 필터링해 알림 발송 신뢰도 개선

S3 파일 정합성 개선

Root Cause:

  • 파일 업로드는 DB 저장보다 먼저 S3에서 성공할 수 있음
  • DB 저장 과정에서 예외가 발생하면 DB 트랜잭션은 롤백되지만, S3에 이미 업로드된 파일은 자동으로 삭제되지 않음
  • DB에는 참조 데이터가 없지만 S3에는 사용되지 않는 파일이 남는 정합성 문제가 발생

S3 Flow

FLOWS3 업로드 완료
FLOW파일 Key 반환
HANDLER이벤트 등록
TRIGGER트랜잭션 실패
BRANCHAFTER_ROLLBACK
ACTIONS3 파일 자동삭제

Solution Steps:

  1. S3 업로드는 DB 트랜잭션과 별도로 동작하는 외부 저장소라는 점을 원인으로 파악
  2. S3 업로드 성공 직후 업로드된 file key를 S3RollbackEvent에 담아 이벤트 등록
  3. DB 저장 로직은 기존 트랜잭션 흐름 안에서 처리
  4. DB 트랜잭션 실패 시 @TransactionalEventListener의 AFTER_ROLLBACK 단계에서 S3 파일 삭제
  5. 예외 발생 위치가 아닌 트랜잭션 결과를 기준으로 보상 작업이 실행되도록 구성

Result:

  • DB 롤백만으로는 S3 파일이 되돌아가지 않는 문제를 보완
  • DB 저장 실패 시 업로드된 S3 파일을 자동 삭제해 DB 데이터와 외부 저장소 간 정합성 유지
MODULE: 03. Workflow Automation
03

에이전트 기반 개발 워크플로 개선

△ Existing Issues

  • PR 작성, 커밋 메시지 작성, 코드 리뷰 준비처럼 반복되는 개발 운영 업무가 많음
  • 작업자마다 작성 기준이 달라질 경우 협업 품질이 흔들릴 수 있다고 판단

⚑ Approach

  • Gemini CLI가 프로젝트 컨텍스트를 일관되게 이해할 수 있도록 GEMINI.md를 작성
  • 반복 업무를 하나의 범용 AI에게 맡기기보다 목적에 따라 역할별 에이전트를 분리

Implementation:

  • code-reviewer 에이전트: 코드 변경 사항을 검토하는 역할로 분리하고, 보안 취약점, 성능 문제, N+1 쿼리 가능성, 예외 처리 방식, 프로젝트 컨벤션 준수 여부를 중점적으로 점검하도록 정의
  • project-manager 에이전트: 구현 세부사항보다 아키텍처 방향, 기능 우선순위, 기술 선택의 타당성을 검토하는 역할로 분리하고, 복잡한 기능 구현 전 데이터 흐름과 도메인 책임을 먼저 점검
  • PR 작성 컨벤션: git diff, 커밋 히스토리, 변경된 DTO/Service/Exception 구조를 분석해 PR 초안을 만들 수 있도록 규칙을 정의하고 Summary, Changes, Test Scenarios 형식을 고정
  • 커밋 메시지 컨벤션: 변경 파일과 작업 의도를 기준으로 커밋 메시지 초안을 작성할 수 있도록 형식을 정의하고 type, subject, issue number, 변경 클래스별 상세 내용을 포함하도록 구성

Result

  • 반복적인 문서 작성 시간을 줄이고, PR과 커밋 메시지의 형식을 일관되게 유지
  • 코드 리뷰 시 보안, 성능, 예외 처리, 컨벤션 준수 여부를 빠르게 점검할 수 있는 개발 흐름 구성
